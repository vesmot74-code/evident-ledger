# Critical: signature persistence investigation

Date: 2026-07-23  
Status: **Root cause confirmed — FIXED** (`fix(ledger): persist signature on legacy /events path`)  
Environment (diagnosis): live pilot process pid 4500, DB `evident_ledger_pilot_11_6`

---

## Root cause

CLI `evident commit` calls **`POST /events`** (legacy), **not** `POST /v1/events`.

| Layer | Code |
|---|---|
| CLI | `src/client.rs` → `POST {base_url}/events` |
| Router | `src/main.rs` → `.nest("/events", api::events::router(…))` |
| Handler | `src/api/events.rs` → `submit_event(…)` |
| Ledger | `src/service/ledger.rs` |

### Mechanism (pre-fix)

1. **`insert_event_in_tx`** inserted with hardcoded empty signature (`.bind("")`).
2. **`submit_event`** committed **before** signing.
3. **`finalize_event_submission`** signed only into the HTTP JSON response.
4. No `persist_event_signature` on the legacy path.

Asymmetry introduced in **`bb43af7`** (v1 persist only).

### Fix applied

`submit_event` now: insert placeholder → `proof_snapshot_at_event` + **`persist_event_signature`** (same helpers as v1) → **`tx.commit()`** → if `ProofStatus::Anchored`, `on_proof_anchored` → finalize response using the **same** snapshot signature bytes (response == DB).

## Path comparison (`/events` vs `/v1/events`) after fix

| Step | Legacy `POST /events` | `POST /v1/events` |
|---|---|---|
| Insert placeholder `signature=""` | Yes (`insert_event_in_tx`) | Yes (shared) |
| Server signature | `proof_snapshot_at_event` / `ServerSigner::sign_root` | Same helpers |
| Persist signature | `persist_event_signature` **before** commit | Same |
| Identity signature / PoP | Optional columns on `SubmitEventRequest`; **no** PoP challenge validation on this path | Validated via `IdentitySigningService` when present |
| Commit | After persist | After persist + idempotency record |
| Public materialization | `on_proof_anchored` if `Anchored` (same gate) | Same gate via `materialize_public_proof_after_anchor` |
| TSA | `stamp_chain` after commit in finalize | `post_commit_tsa` after commit |
| Client JSON | Full proof object (CLI) | Compact `{event_id, proof_status, …}` |

**ServerSigner:** both handlers receive `state.signer.as_ref()` from the single `Arc<ServerSigner>` loaded in `main.rs` via `ServerSigner::load_or_create(&config.signing_key_path)`.  
Confirmed: **legacy and v1 signature bytes are generated by the same ServerSigner instance/path.**
