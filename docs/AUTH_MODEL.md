## 1. API Key Lifecycle

### Format

```
ev_<secret>
Example: ev_a83f91c2...
```

- **Secret:** 32 hexadecimal characters (128 bits of entropy)
- **Prefix:** `ev_` — identifies the credential as an Evident Ledger API key (metadata, not secret)

### Generation

The secret is generated by a cryptographically secure random number generator.

### Storage

Only a hash of the **secret portion** is stored in `api_keys.key_hash`:

```
key_hash = SHA-256(secret)
```

The `ev_` prefix is **not** included in hashing — it is metadata and is not secret material.

**Legacy lookup (pre-Stage 8.1 keys):** API keys created before the `ev_<secret>` format (dev seeds, manual inserts, early integration tests) did not use the secret-only hash. Authentication **MUST** accept them until explicitly retired:

```
key_hash = SHA-256(full_key_as_presented_in_X-API-KEY)
```

A credential is treated as **current-format** only when it matches `ev_` + exactly 32 hexadecimal characters; all other shapes use legacy full-string hashing. New keys created via `POST /accounts/register` or `POST /accounts/api-keys` always use the current format.

**Removal policy:** legacy lookup is a **compatibility path**, not part of the long-term contract. It may be removed only after a dated update to this document and confirmation that no production accounts rely on non-`ev_` keys.

**Rationale:** a 128-bit secret makes brute force infeasible. SHA-256 is sufficient for this threat model; bcrypt/argon2 would add per-request latency without meaningful security gain for high-entropy bearer tokens.

The original key in plaintext **must not** be stored.

### Return policy

The full API key (`ev_<secret>`) is returned **exactly once** — at creation.
Re-fetching the same key is impossible.
If a key is lost, a new key must be created.

### Revocation

`revoked_at` — when set, the key stops working immediately.
Revocation is irreversible.

### Display / listing

Only a **prefix** (e.g. `ev_a83f…`) and metadata are shown:

- `created_at`
- `revoked_at` (if revoked)

**Legacy rows (pre-Stage 8.1):** the plaintext prefix cannot be recovered from `key_hash` alone. Such rows store an internal sentinel (`legacy:no-prefix` in the database) and list as:

```
legacy key — prefix unavailable
```

This is not a secret fragment and must not be mistaken for an `ev_…` prefix. Operators should rotate legacy keys to current-format keys when convenient.

`last_used_at` is **not** in the current schema; it may be added in a future migration if needed.

The full key is never displayed after creation.

## 2. Account Model

**Current schema (Phase 8.1):**

| Field | Role |
|-------|------|
| `account_id` | Owner of all account-scoped resources |
| `email` | Unique account identifier |
| `tariff_plan_id` | UUID reference to `tariff_plans` |
| `subscription_status` | `none` / `active` / `past_due` / `canceled` |

**Deferred to Stage 8.3 (Dashboard):** `password_hash`, web sessions, and cookie/JWT login flows are **not** part of Phase 8.1.

## 3. Authentication Flows

### Public account bootstrap

**Endpoint:** `POST /accounts/register`

**Authentication:** None (public)

**Request body:**

```json
{
  "email": "user@company.com",
  "company_name": "Pharma Corp"
}
```

`company_name` is optional.

**Validation:**

- `email` — valid email format
- `company_name` — non-empty when provided

**Duplicate email handling:**

If the email is already registered, the server returns **`409 Conflict`**.

This is the **only documented exception** to Security Invariant 16 (non-leak):

- `409` reveals that the email is already registered
- **Rationale:** email is treated as a public account identifier; silent success would mislead the user into believing registration succeeded
- Other errors (invalid email, internal failure) **must not** reveal whether an account exists

**Actions:**

1. Verify email is not already taken
2. Create account with `tariff_plan_id` = free plan
3. Create the first API key
4. Return account metadata and the one-time API key

**Response:**

```json
{
  "account_id": "uuid",
  "api_key": "ev_a83f91c2...",
  "tariff_plan_id": "<uuid>",
  "plan_name": "free",
  "created_at": "2026-07-18T14:40:18Z"
}
```

The API returns `plan_name` (via join with `tariff_plans`) for client convenience, not only `tariff_plan_id`.

---

### Protected account endpoints

**Endpoints:**

```http
GET    /accounts/me
GET    /accounts/api-keys
POST   /accounts/api-keys
DELETE /accounts/api-keys/{id}
```

**Authentication:** `X-API-KEY` (bearer token)

All `/accounts/*` endpoints except `POST /accounts/register` use the same authentication mechanism as `/v1/*`.

### Future — Stage 8.3 (Dashboard)

```http
POST /auth/login
POST /auth/logout
```

Session-based access (cookie or JWT). Requires `password_hash` in the account schema (added in Stage 8.3).

## 4. Ownership Boundary

`account_id` is the owner of:

- chains
- events
- proofs
- API keys
- usage records
- (future: backups, subscriptions, billing)

**Rule:** every resource belongs to exactly one account. Cross-account access is forbidden (`404` non-leak).

See also [SYSTEM_CONTRACT.md](../SYSTEM_CONTRACT.md) §17 (Ownership Boundary).

---

## 5. Security Invariants

These invariants are normatively listed as items **12–16** in [SECURITY.md](../SECURITY.md) §2.5:

12. API keys are bearer credentials with the same security requirements as passwords.

13. API keys are never stored in plaintext.

14. Revoked API keys are rejected immediately.

15. Account is the ownership boundary for all resources.

16. Authentication layer **MUST NOT** leak account existence via timing or error messages — except the documented `409 Conflict` on duplicate email at `POST /accounts/register` (see §3 above).
